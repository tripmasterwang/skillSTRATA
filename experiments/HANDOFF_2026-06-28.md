## 1. 方法理解（今天讨论达成的共识）

- **三层结构（trace / capability / governance）是「载体」，不是核心**。核心是两套机制：
  1. **离线 curate**：从空图开始，逐轮 `rollout→distill→integrate(INSERT/MERGE)→split(SPLIT)→validation_gate→checkpoint`，no-grad / no-RL。
  2. **推理时使用**：域内 ROUTE（最小可执行子图）+ 域外 TTA（从 trace 层共现证据当场合成临时技能）。
- **「为什么是三层不是五六层」**：按「操作对象的本体类型」切——事实(trace)/能力(capability)/管能力的规则(governance)，恰好三类。粒度(atomic/functional/plan)是层内细分（`Granularity`），不算新层。三层部分继承自 G-Memory，有约定成分（审稿人可能质疑）。
- **方法的两个命门（必须在论文里守住）**：
  1. `distill_fn`（总结器）是**黑盒**、且是所有方法共用的——贡献不能赖在"我总结得好"，全靠消融证明"同样的料，组织得好 vs 堆一坨差很多"。
  2. 三层必然性偏弱，可能被说成"凑 G-Memory"。
- **TTA（测试时合成）当前是配角**，brief 里标"非主卖点"。用户倾向把它的"测试时自造技能"故事讲得更重，但**证据偏弱**（覆盖只补回 ~54%、token +88%），要扶正必须先把它的实验做硬。**这是一个待定的定位选择。**

## 5. 消融实验设计（深刻分析方法的 5 个解剖轴）

| 轴 | 回答 | 设计 |
|---|---|---|
| ① 必要性×充分性 | 每部件单独值多少 | LOO + LOI 双向 |
| ② 进化动力学 | 逐轮在进化什么 | per-round 算子分布 + 库成分 + 门 accept/reject + 负迁移翻转随轮 |
| ③ 路由机制 | 省 token/涨分来自哪 | routing precision + 依赖闭包补回前置频次 + loaded token |
| ④ 治理因果 | 门+checkpoint 挡掉/修好了什么 | 被拒批次特征 + verify-loop 修复率 + case study |
| ⑤ 跨 harness 不变性 | 图能否跨脚手架迁移 | 同一 from-zero 图路由到 4 harness |

**可控旋钮（真实代码里）**：
- 进化期：MERGE(`integrate_fragments` sim_threshold) / SPLIT(`curate(do_split=)`) / 验证门(`validation_gate`) / checkpoint(`checkpoint_fn`) / heat归因(`_attribute_heat`)
- 推理期：`SKILLSTRATA_ROUTER=graph/flat/full` / `SKILLSTRATA_VERIFY_LOOP=0/1`
- 进化轮数：`ROUNDS`

**三个 RQ → 图**：
- RQ1：图1 进化曲线 + 图2 算子分解堆叠柱（"长技能→治理→挂门"三阶段假设，待验证）
- RQ2：图3 部件贡献条形图（LOO+LOI，区分进化期/推理期），找最关键部件
- RQ3：图5 能力雷达（6 维：token膨胀/负迁移/OOD/有害技能/长程稳定/冷启动）+ 待改善二分表
